# Drizzle migrations — notes for the v0.1 underwriting tables

## Workflow

This project previously used `drizzle-kit push` (declarative diff against the live DB) rather than versioned migrations. The migration in this folder (`0000_underwriting_v01.sql`) was generated by running `npm run db:generate` after adding `underwritingResults` + `softCommitments` to [lib/db/schema.ts](../lib/db/schema.ts).

**Important:** `0000_underwriting_v01.sql` includes `CREATE TABLE` for all 13 tables, not just the new two — because there was no prior versioned baseline. If you apply it directly to the existing Neon production DB you will get duplicate-table errors.

## Two ways to apply on production

### Option A — declarative push (recommended)
```bash
DATABASE_URL=... npm run db:push
```
Drizzle diffs the live DB against the schema and applies only the deltas: the new `underwriting_status` enum, the `underwriting_results` table, the `soft_commitments` table, plus their indexes and FKs.

### Option B — incremental SQL (manual)
Apply `apply_v01_underwriting_only.sql` (extracted below) directly via psql or the Neon console. This file contains only the new objects, safe against an existing DB.


## Rollback

```sql
DROP TABLE "soft_commitments";

```

## Schema notes (why these fields)

- **`deal_id` (not project_id alone)** — the auditor starts data entry before any on-chain Solana project exists. `deal_id` is the human-readable URL slug (`/lender/{deal_id}`). When the on-chain project is later created, set `project_id` to link it.
- **`audit_inputs_json` + `prediction_json`** — raw blobs of the FastAPI `PredictRequest` and `PredictResponse`. The typed numeric columns mirror the most-queried fields for index/JOIN speed; the JSONB blobs preserve everything else for forensics.
- **`status` enum** — tracks the auditor→lender→Day-30 lifecycle. Drives the `/lender/[deal_id]` page state.
- **`p5_violated_flag` + `point_estimate_delta_pct` + `reconciliation_passes`** — populated by `src/reconcile_predictions.py` once realized Day-30 savings are entered. The MVP success criterion compares these aggregate stats vs ±15% / ±20% thresholds.
- **`unique(deal_id, ecm_id)`** — each ECM within a deal can only have one underwriting record (re-submission updates in place).
